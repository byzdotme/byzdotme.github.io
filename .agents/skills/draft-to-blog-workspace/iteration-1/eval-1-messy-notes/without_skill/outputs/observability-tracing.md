# 分布式链路追踪：从 OpenTelemetry 到全链路可观测

在单体应用时代，查问题的方法很简单：找到那台服务器，看日志、查指标。但在微服务架构下，一个用户请求可能要经过网关、认证、订单、支付、通知等十几个服务。问题出现时，你甚至不知道请求去了哪个服务，更别说每个服务花了多少时间。

分布式链路追踪（Distributed Tracing）就是为了解决这个"盲人摸象"的问题而生的。

## 1. 核心概念

在说具体技术之前，先理清几个基础概念：

- **Trace：** 一次完整请求的端到端链路。从用户点击到最终响应，整条路径就是一个 Trace。
- **Span：** Trace 中的一个操作单元。一个 Span 代表一个服务内部的一段处理逻辑（比如"查询数据库"或"调用订单服务"）。Span 可以嵌套，形成父子关系。
- **SpanContext：** 跨服务传递的上下文信息，包含 TraceId、SpanId、父 SpanId 等。通过 RPC 或 HTTP Header 传递，是串联不同服务的关键。

如果拿打电话来类比：Trace 是整个通话，Span 是通话中每一句话。没有 Trace，你知道有人在说话，但不知道这是谁在说话、和谁说话、说了多久。

**小结：** Trace 和 Span 是分布式追踪的两个核心抽象。Trace 是"面"，Span 是"点"，SpanContext 是线，三者共同构建了请求的全景图。

## 2. OpenTelemetry：事实标准

OpenTelemetry（简称 OTel）是 CNCF 的孵化项目，目前已经是分布式追踪的事实标准。它不仅是追踪，而是整合了**度量（Metrics）** 和**日志（Logs）**，实现可观测性的三位一体。

### 架构组成

OTel 的架构分为四层：

1. **API：** 定义数据模型和采集接口，语言无关。无论你是 Java、Go 还是 Python，API 的概念是一致的。
2. **SDK：** 具体语言的实现，负责上下文传播、采样、数据处理等。
3. **Collector：** 独立的 Agent 组件，接收、处理、导出 OTel 数据。Collector 支持数据过滤、采样、批处理和重试，是整个管道的枢纽。
4. **Exporter：** 将数据发送到后端存储（Jaeger、Zipkin、Prometheus、Tempo 等）。

### 自动埋点 vs 手动埋点

OTel 提供了两种埋点方式：

- **自动埋点（Instrumentation）：** 通过字节码增强（Java Agent）或中间件钩子，自动拦截 HTTP、gRPC、数据库等标准库的调用。零代码侵入，开箱即用。
- **手动埋点：** 在业务代码中显式创建 Span，适合自定义业务逻辑的追踪（比如"计算运费"这个操作花了多久）。

在实践中，两者通常是配合使用的：自动埋点覆盖基础设施层面的调用链，手动埋点补充业务语义。

**小结：** OTel 的自动埋点降低了落地门槛，手动埋点提供了精细控制的入口。先用自动埋点跑起来，再逐步补充关键业务的手动埋点。

## 3. 采样策略

如果对每个请求都做全量追踪，最边缘的服务每秒可能产生数万个 Span。数据量巨大，磁盘和网络都扛不住。所以需要采样。

常见的采样策略有三种：

**固定采样率（Head-based Sampling）：** 最简单的方案。在请求开始时，按固定比例（比如 10%）决定是否采。优点是实现简单、开销小；缺点是采样决策在请求开始时做出，无法根据请求结果（成功或失败）调整权重。可能导致少量有价值的失败请求没有被采集到。

**自适应采样：** 基于流量特征动态调整采样率。正常流量少采，异常流量多采。比如延迟突增时自动提高采样率。实现复杂度适中，推荐作为线上默认策略。

**头部采样（Head-based） vs 尾部采样（Tail-based）：**
- 头部采样：请求到达时立即决定是否追踪。优点是不需要缓存 Span，缺点是决策信息不足。
- 尾部采样：请求结束后再决定是否保留该 Trace。优点是可以看到完整的请求信息再决策（比如只看执行慢或失败的 Trace），缺点是 Collector 需要缓存所有 Span 直到请求结束，对内存和延迟有要求。

对于大多数业务场景，**自适应头部采样 + 对已知错误强制全采** 是一个性价比较高的组合。尾部采样更适合大规模、对数据完整性要求极高的场景。

**小结：** 采样是 Tracing 落地必须面对的问题。不要上来就追求尾部采样，先用好固定采样率和自适应采样，等规模上来后再评估是否需要尾部采样。

## 4. 后端存储选型

采集到数据之后，需要找个地方存起来并提供查询能力。几个主流方案对比：

- **Jaeger：** 老牌分布式追踪系统，功能全面，支持多种存储后端（Elasticsearch、Cassandra、Kafka）。社区成熟，文档丰富。不过现在 Jaeger 也在全面拥抱 OpenTelemetry 协议，建议直接使用 OTel 采集，再通过 Jaeger 存储和展示。
- **Zipkin：** 更早期的方案，Twitter 开源。现在使用场景较少，新项目不推荐。
- **Grafana Tempo：** 新一代追踪存储系统，专为 OTel 设计。最大的特点是**不需要专门的存储集群**，直接使用对象存储（S3、GCS、Azure Blob）。成本远低于 Jaeger + ES 的组合。与 Grafana 生态深度集成，一套 Grafana 实例可以同时查 Tracing、Metrics 和 Logs（通过 Loki），体验统一。
- **ELK：** 用 Elasticsearch 存储 Trace 数据不是不行，但没有专门针对 Trace 数据模型做优化，查询体验差，不推荐。

**小结：** 如果从零开始搭建，推荐 **OTel Collector + Grafana Tempo + Grafana** 的组合，成本低、集成好、符合技术趋势。如果已有 Jaeger/ES 基础设施，继续使用也没有问题。

## 5. 实践经验

最后几个在落地过程中容易忽略的要点：

- **TraceId 要贯穿整个链路。** 从网关层生成，通过 HTTP Header 或 RPC 元数据一路传递到最底层的数据库查询。任何一级的丢失都会导致链路断裂。
- **Span 命名规范。** 每个 Span 的 Name 代表一个操作类型，命名应当统一规范（如 `HTTP POST /api/orders`），不能太随意。
- **Attribute 不要太多。** 每个 Span 可以附带键值对属性（Attribute），但过多的属性会影响序列化性能和存储开销。只保留对排查有实际帮助的信息。
- **异步操作记得传递 Context。** Go 的 goroutine、Java 的线程池、Node.js 的异步回调中，必须显式传递 Trace Context，否则链路会在异步边界断开。
- **非 HTTP 协议的追踪。** OTel 不仅支持 HTTP，还支持 gRPC、消息队列（Kafka/RabbitMQ）、数据库调用等协议。不同协议的上下文传播方式不同，需要分别配置。

## 总结

分布式链路追踪是微服务架构下排查问题的"导航仪"。从 Trace/Span 的基础概念，到 OpenTelemetry 的架构设计，再到采样策略和存储选型，环环相扣。

对于刚接触 Tracing 的团队，推荐的落地路径是：**先用 OTel 自动埋点跑通基础链路的追踪，配合 10% 固定采样率控制数据量，后端选择 Jaeger 做查询展示。等团队熟悉了 Tracing 的使用方法和价值后，再逐步引入手动埋点、自适应采样和 Tempo 等更高级的存储方案。**
