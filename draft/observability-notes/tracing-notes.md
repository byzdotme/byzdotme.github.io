# Tracing 笔记

## 分布式追踪是啥

微服务架构下，一个请求可能经过十几个服务。如果没有追踪，根本不知道请求经过了哪些服务、每个服务花了多少时间。

核心概念：
- Trace：一次完整的请求链路
- Span：链路中的一个操作（可以嵌套）
- SpanContext：跨服务传递的上下文，包含 TraceId、SpanId 等

## OpenTelemetry

otel 是 CNCF 的项目，现在是分布式追踪的事实标准。它不只是 tracing，还包括 metrics 和 logging，三位一体。

### 架构
- API：定义接口
- SDK：具体实现
- Collector：接收、处理、导出数据
- Exporter：把数据发到后端（Jaeger、Zipkin、Prometheus...）

### 自动埋点 vs 手动埋点
自动埋点适合 HTTP/gRPC/数据库等标准库，零代码侵入。
手动埋点适合业务逻辑（比如"下单"这个操作花了多久）。

## 采样策略

如果所有请求都记录 tracing，数据量太大了。需要采样。

常见的采样策略：
1. 固定采样率：比如 10% 的请求
2. 自适应采样：正常请求少采，异常请求多采（更合理）
3. 头部采样 vs 尾部采样：
   - 头采样：在请求开始时决定是否追踪（简单但有信息损失）
   - 尾采样：请求结束后再决定（可以看到完整信息，但需要缓存所有 span）

尾采样更好但是实现复杂，要缓存所有 span 等请求结束再决策。

### 一个错误观点
~~有人说 opentelemetry 只支持 HTTP 协议~~（这是错的，还支持 gRPC、UDP 等）

## 后端选择
- Jaeger：老牌，功能全
- Zipkin：更老，现在用的少了
- Grafana Tempo：新一代，存对象存储，成本低
- 直接用 ELK 做 tracing 也不是不行（但是不好用）

Tempo 是个好选择，它不需要专门的存储集群，直接存 S3/GCS，成本低很多。而且和 Grafana 集成好。

## 实践经验
- traceId 要贯穿整个链路，从网关到数据库查询
- span 的命名要规范，不能太随意
- 别给每个 span 加太多 attribute，会影响性能
- 异步操作记得传递 context
