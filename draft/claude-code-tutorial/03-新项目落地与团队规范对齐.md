# 03 · 新项目落地与团队规范对齐

> 场景：你刚 `git clone` 了一个仓库，用 CC 打开。这个仓库可能是**一片空白**（新项目），也可能是**迭代了 N 年的屎山**，甚至用了一堆公司内部闭源框架和中间件。
>
> 这篇教你怎么做 **CC onboarding**，让它写出来的代码真的符合团队口味。

## 0. 统一思路：三类信息要分别承载

团队规范通常包含三种"颗粒度"的信息，**不要混在一个 CLAUDE.md 里**：

| 信息类型 | 承载物 | 原因 |
|---------|-------|------|
| **铁律（必须遵守）** | `CLAUDE.md`（项目根 + 子目录） | 常驻上下文，每次必看 |
| **SOP（特定场景才需要的做法）** | Skills | 按需加载，不污染上下文 |
| **参考资料（偶尔查阅的背景）** | `docs/` 目录 + 让 CC 按需 `@` 引用 | 大文件不适合常驻 |

一个例子：

- "本项目用 pnpm 不用 npm" → CLAUDE.md（铁律）
- "新建一个 DDD 聚合根的 12 步" → Skill `ddd-aggregate-new`
- "公司 RPC 框架 v3 到 v4 的迁移指南" → `docs/frameworks/rpc-v4-migration.md`

## 1. 空项目 / 新项目的 Day-1 SOP

假设你刚 `mkdir` 或 clone 了一个空仓。按这个顺序做：

### Step 0 · 把鉴权搞清楚（容易被忽略，但关系到计费和团队接入）

CC 有两条独立鉴权路径，**不要混用**：

| 路径 | 命令 | 用在哪 | 计费 |
|------|-----|-------|------|
| Claude 订阅（Pro / Max / Team） | `claude auth login` | 个人/团队日常交互式使用 | 走订阅额度 |
| API key | 设置 `ANTHROPIC_API_KEY` 或 `claude auth login --console` | CI / 长 cron / 不能交互 | 按 token 计费 |
| 长期 OAuth token | `claude setup-token` 一次性生成，打到 CI secret | CI 里走订阅额度 | 走订阅 |

`claude auth status` 看当前登录状态。团队 onboarding 第一步就讲清楚"个人订阅 token 不要进 git/CI 共享"，避免被同事不小心耗光额度。

### Step 1 · 先决定骨架再写代码

按 **Shift+Tab 两次**（或键入 `/plan`）切到 plan mode，然后：

> "我要用 <技术栈> 搭一个 <业务>。团队规范是 <DDD/TDD/Clean Architecture/…>。
> 先做目录骨架和最小可运行 skeleton，不要实现具体业务。think harder。"

**让它先给计划**，你 review 技术选型、目录结构、工具链后再 accept。

### Step 2 · 锁定团队约定到 CLAUDE.md

让 CC 自己跑 `/init` 生成一份雏形，然后**你手工扩写**。推荐模板：

```markdown
# <项目名> · Claude 指南

## 项目速览
- 业务：<一句话>
- 架构范式：DDD + Hexagonal / Clean Architecture
- 语言/运行时：TypeScript 5.x / Node 20
- 关键依赖：<内部框架 A v3>、<中间件 B>

## 绝对铁律
- 包管理器：**只能用 pnpm**，禁止 npm/yarn
- 测试：TDD 流程——先写失败用例，再实现（见 `.claude/skills/add-test-tdd/`）
- 日志：一律用 `@company/logger`，禁止 console.log
- 配置：一律通过 `@company/config`，禁止直接读 process.env
- 时间：服务端一律 UTC；禁止用 `new Date()`，改用 `@company/clock`
- 错误：对外错误统一 `AppError`，禁止抛裸 Error
- 提交：Conventional Commits + 中文描述；不加 co-authored-by

## 目录契约
- `src/domain/**`：领域层，不得 import 任何 infra/web
- `src/application/**`：应用服务/用例
- `src/infrastructure/**`：适配器（DB、外部 API、MQ）
- `src/interfaces/**`：HTTP/RPC/CLI 入口
- 测试与被测代码同目录 `*.test.ts`

## 命令速查
- `pnpm dev`、`pnpm test`、`pnpm test:int`、`pnpm lint`、`pnpm typecheck`
- **每次改动后 CC 必须自己跑 `pnpm typecheck && pnpm test:affected`**

## 风格偏好
- 函数优先，class 仅用于 Entity / ValueObject / 特定模式
- 返回 Result<T, E> 而不是抛异常（除非真的是不可恢复）
- 不写没必要的注释；注释只解释"为什么"

## Skills 索引
- `.claude/skills/ddd-aggregate-new/` 新建聚合根
- `.claude/skills/internal-rpc-handler/` 创建 RPC 接口
- `.claude/skills/add-feature-flag/` 加特性开关

## 参考文档（按需 @ 引用，不要默认加载）
- `docs/architecture.md` 架构总览
- `docs/frameworks/rpc.md` 内部 RPC 框架
- `docs/frameworks/orm.md` 内部 ORM
```

> **铁律部分尽量用"禁止 / 必须"句式，不要用"建议 / 尽量"**——Agent 对硬约束执行力更强。

### Step 3 · 配基础工具链 Hook

让每次 CC 改代码后自动 typecheck + lint，挡掉低级错误（见 02 篇 Hooks 章节）。

### Step 4 · 建一个"团队风格" Skill 最小集

最起码建议先起三个：

1. `.claude/skills/add-test-tdd/` —— 强制 TDD 节奏
2. `.claude/skills/new-module/` —— 按 DDD 目录结构脚手架
3. `.claude/skills/commit-style/` —— 写出符合团队的 commit message

## 2. "屎山"项目 / 存量大仓的 onboarding SOP

这是真正的挑战。CC 不认识你们公司的内部框架，不知道你们多年沉淀的 "unwritten rules"。按下面这套 6 步流程做。

### Step 1 · 让 CC 做一次"考古侦察"

**Shift+Tab 两次**进入 **Plan 模式**（只读），然后**派给内置 Explore subagent**做（它默认走 Haiku 4.5，又快又便宜）：

```
ultrathink. 把下面这份 prompt 派给 Explore subagent 去做，主会话只接收最终报告：

你是新入职的资深工程师。生成一份项目考古报告：

1. 识别语言、构建系统、包管理器、测试框架、lint 工具
2. 识别目录结构及每个顶层目录的职责（看命名、看里面文件、看 import 关系）
3. 识别使用到的公司内部包（所有以 @company/ 或 @<我们命名空间>/ 开头的依赖），
   列出每个包出现的文件数、典型用法片段（各给 2 个示例位置）
4. 识别明显的"历史沉淀"：
   - 同一功能有多种实现的地方（例如同时存在 axios / 内部 http client）
   - 明显已废弃但还没清理的代码（deprecated 注释、TODO）
   - 风格断层（某个目录风格和其它不一样）
5. 识别测试策略：单测/集成/e2e 的比例和分布
6. 识别危险区域：循环依赖、上帝文件(>1000 行)、热点文件(git log 改动最多)
7. 输出一份 ARCHAEOLOGY.md 草稿

不要修改任何代码。不要猜，你没把握的地方明确标 UNKNOWN。
```

**为什么先派给 Explore**：脏活累活（grep、read 大量文件）跑在子代理里**只把摘要返回**，主会话上下文不被海量源码占满。如果仓库很大（多 service / 多 package），可以用 `/batch` 派 5–10 个 Explore subagent 各盯一块，并行考古，最后由主代理合并报告。

### Step 2 · 根据侦察报告分层写 CLAUDE.md

屎山仓库**不要只写一个根 CLAUDE.md**——会越写越长。使用分层策略（详见 04 篇）：

```
/CLAUDE.md                       # 项目级铁律 + 目录地图
/backend/CLAUDE.md               # 后端特有规范
/backend/billing/CLAUDE.md       # 计费模块特有的踩坑记录
/frontend/CLAUDE.md              # 前端特有规范
/packages/legacy-core/CLAUDE.md  # 遗留代码警告："这里别重构，只加不改"
```

子目录的 CLAUDE.md 会在 CC 进入该目录上下文时**自动追加**到可见记忆中。

### Step 3 · 给内部框架 / 中间件 "教教科书"

这是屎山仓最关键的一步：**模型不认识 `@company/xxx`，必须你来教它。**

三种方式，按 ROI 从高到低：

#### 方式 A（强推）· 为每个内部框架写一个 Skill

```
.claude/skills/
├── internal-rpc/
│   ├── SKILL.md              # 何时用 / 不用；核心概念；5 个常见错误
│   ├── templates/handler.ts  # 新 handler 模板
│   └── references/
│       └── full-api.md       # 详细 API（按需加载）
├── internal-orm/
└── internal-mq/
```

每个 SKILL.md 不要超过 200 行，把冗长的 API 文档放 `references/`。description 一定要写清楚"遇到 import '@company/rpc' 或用户说 'handler' 时触发"。

#### 方式 B · 把框架文档纳入 `docs/frameworks/`

如果框架本身有内部 docs，直接复制/同步一份到仓库 `docs/frameworks/`。在根 CLAUDE.md 里**只列索引**，告诉 CC：

```markdown
## 内部框架索引（按需 @ 引用，不要默认全部加载）
- `docs/frameworks/rpc.md` 内部 RPC
- `docs/frameworks/orm.md` 内部 ORM
- `docs/frameworks/mq.md` 内部消息队列
```

这样 CC 在需要用某个框架时，你（或它自己通过描述匹配）会 `@docs/frameworks/rpc.md` 加载，不会污染常态上下文。

#### 方式 C · 用"示例优先"的 Skill

如果内部框架文档缺失、只能看代码——让 CC 自己从代码里归纳：

```
ultrathink. 扫描 packages/ 下所有使用 @company/rpc 的文件，归纳出：
1. Handler 的典型骨架（给一个最小完整示例）
2. 错误处理的惯用法
3. 中间件注册的方式
4. 常见反模式
输出为 .claude/skills/internal-rpc/SKILL.md 草稿。
```

然后你再人工 review、修订。**让 CC 读代码自学 + 你审核**，比你自己从零写文档快得多。

### Step 4 · 用"学习示例"锚定风格

CC 模仿人类代码风格的能力**非常强**，但你要显式告诉它学谁。在 CLAUDE.md 里写：

```markdown
## 风格锚点
当实现新模块时，请以下面这些文件为风格范本（命名、分层、注释、错误处理）：
- 好样本：`src/modules/order/OrderService.ts`
- 好样本：`src/modules/inventory/` 整个模块
- 反样本（不要学）：`src/modules/legacy-pricing/`（历史遗留，即将废弃）
```

**这比写 100 条风格规则都有效**，因为代码里已经把所有隐含规范（命名风格、日志调用、错误类型、注释密度）都固化了。

### Step 5 · 为 TDD / DDD 等方法论写硬约束

如果团队强制 TDD：

```markdown
## TDD 强制流程
写任何业务代码前，你必须：
1. 先创建 `*.test.ts`，写出覆盖验收条件的失败测试
2. 跑 `pnpm test <path>`，确认测试是失败的（红）
3. 才能写实现代码，让测试变绿
4. 重构（如需要）

跳过 step 1-3 直接写实现是 **严重违规**，必须重做。
```

如果团队强制 DDD：

```markdown
## DDD 约束
- 领域层不得 import 任何 infrastructure / interfaces / framework 代码
- Entity 只能通过 Repository 持久化，禁止在应用层直接拼 SQL
- 业务不变量必须用 ValueObject 或 Entity 方法表达，禁止在 Service 里散落校验
- 每个聚合根必须有一个 `*.invariant.test.ts` 证明不变量
```

这类铁律写进 CLAUDE.md 比任何 Skill 都优先。

### Step 6 · 首轮小任务校准

不要直接让 CC 上手做大需求。**用一个小任务做"风格校准"**：

1. 挑一个简单 bug 或重构。
2. 让 CC 做（走 plan 模式）。
3. 你对它产出的代码做严格 diff review，凡是不符合团队风格的都 → **反馈进 CLAUDE.md 或对应 Skill**。
4. 再让它做第二个小任务，迭代到它产出基本符合预期。

这一步走 1-2 天，后面几个月都受益。

## 3. 团队规范常见条目速查（抄作业）

下面是一些在资深团队里常见、但很容易被 CC 忽略的规范，**直接复制到你的 CLAUDE.md 里选用**：

```markdown
## 日期/时间
- 禁止 `new Date()`、`Date.now()`；使用 `clock.now()` (@company/clock)
- 所有持久化/传输的时间一律 ISO8601 UTC
- 展示层才做时区转换

## 幂等性
- 所有对外写接口必须支持 idempotency key
- MQ 消费者必须幂等（用业务 id + 版本号）

## 可观测
- 每个入口（controller / handler / consumer）起始处埋 trace
- 错误日志必须包含：traceId、业务 id、可定位的 error code
- 业务指标用 `@company/metrics`，禁止直连 prometheus 客户端

## 并发
- 禁止在请求期内起 detached promise（不 await 且无错误处理）
- 共享状态一律靠分布式锁或乐观锁，禁止进程内锁（多副本场景）

## 数据库
- 迁移必须双写兼容：先上新 schema，双写，再切读，再删旧字段
- 禁止在 migration 中写业务逻辑修数据；用单独 backfill 脚本

## 特性开关
- 所有新功能通过 `@company/flags` 灰度，默认关闭
- 上线满 N 天后清理开关，留清理 TODO

## 安全
- 禁止 log 任何含 PII 的字段（见 PII 白名单 `docs/security/pii.md`）
- 所有外部输入必须经 schema 校验（zod/io-ts/…）才进业务层

## Git/PR
- 一个 PR 一个关注点，超 400 行 diff 必须拆
- commit message 中文描述为主，首行 <= 50 字
- 禁止 force push 到共享分支
```

## 4. 一个"从零让 CC 符合团队要求"的 Checklist

打印版，对着做：

- [ ] 根目录 `CLAUDE.md`：项目速览 + 铁律 + 目录地图 + 风格锚点 + Skills 索引
- [ ] 关键子模块 `*/CLAUDE.md`：模块特有规则、踩坑记录
- [ ] `.claude/skills/` 至少 3 个：内部框架 × N + 团队方法论 Skill
- [ ] `docs/frameworks/` 同步或归纳内部框架文档
- [ ] `.claude/settings.json` 配好 PostToolUse typecheck/lint hook
- [ ] 配好 `/permissions`：禁止危险 bash、限制只在工作区内
- [ ] 接入 1-2 个 MCP（至少 Git 托管 + 任务系统）
- [ ] 选 2-3 个"好样本文件/模块"写进风格锚点
- [ ] 走一次小任务校准，把发现的偏差回写到 CLAUDE.md
- [ ] 在 `settings.json` 里指定团队默认 `agent`（启动后默认进入哪个主代理）
- [ ] 启用 auto memory（v2.1.59+）作为低优先级补充，第一周末 review 一次记忆文件
- [ ] 提交到 git 并让团队成员也 pull 一份
