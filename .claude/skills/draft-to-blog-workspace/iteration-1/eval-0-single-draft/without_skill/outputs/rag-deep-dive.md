# RAG 深度解析：从检索增强到 Agentic 架构的工程实践

这两年做大模型相关的技术面试，RAG 几乎是必考题。而且面试官的追问越来越细：从"RAG 解决了什么问题"到"你们的 chunk 怎么切的、用什么检索策略、有没有加 rerank、评估指标怎么设计的"——显然，RAG 已经从一个概念验证话题变成了要能讲出工程细节的硬核题目。

本文从 Agent 架构视角切入，系统梳理 RAG 的全链路工程细节，覆盖从数据准备到生产落地的各个关键环节。希望能帮你在面试中从原理到工程、从指标到调优都讲得清楚。

## 一、什么是 RAG，为什么需要它

**RAG（Retrieval-Augmented Generation）**，即检索增强生成。核心思路很简单：在 LLM 生成答案之前，先从外部知识源检索相关文档，作为上下文喂给模型。

这个思路之所以能火起来，是因为 LLM 天然有三个硬伤：

**知识截止**。模型训练数据有截止日期，不知道今天发生了什么。RAG 通过检索实时数据来补上这个缺口。

**幻觉**。LLM 会编造事实，尤其在它不确定的领域。RAG 让模型基于真实检索结果回答问题，大幅降低了捏造的可能。

**领域知识不足**。通用模型对内部文档、专业术语了解有限，RAG 相当于把私有知识库直接接到了模型前面。

当然，RAG 也不是银弹。如果你的场景是数学推理、代码生成或创意写作，本身不依赖外部知识，那 RAG 帮不上什么忙。另外，如果目标是把某种风格或领域术语固化到模型里，fine-tuning 反而是更合适的手段。

**小结：** RAG 的核心价值在于以较低的维护成本，为 LLM 接入实时、私域、可验证的外部知识，弥补了预训练模型在事实性知识上的短板。

## 二、RAG vs Fine-Tuning：什么时候选谁

面试中经常被问到的一个问题：什么时候用 RAG，什么时候用 fine-tuning？我的理解是这样的：

RAG 的优势在**更新成本低**——改一下知识库内容就行，不需要重新训练模型。它适合承载大容量的事实性知识，能到 TB 级，而且天然有引用可追溯，可解释性好。缺点是每次回答多了一步检索，延迟会比纯模型生成高。

Fine-tuning 的优势在于**改变模型的行为范式**——比如让模型学习特定的回答风格、行业术语，或者固化某种任务逻辑。它没有检索这一步，延迟低，但更新成本高（重新训练），知识范围受限于模型容量。

更常见的做法是两者结合：用 fine-tuning 让模型理解领域语言和任务范式，用 RAG 承载具体的事实性知识。

**小结：** 事实性知识交给 RAG，范式/风格交给 fine-tuning，两者互补而非互斥。

## 三、整体架构：从离线到在线

RAG 系统的全链路分为离线和在线两个阶段。

离线阶段负责数据准备：

```
原始文档 → 清洗 → 切分（chunking）→ 向量化（embedding）→ 存入向量库 + 建立索引
```

在线阶段负责检索与生成：

```
用户问题 → 查询改写 → 向量化 → 向量检索 + 关键词检索
                                        ↓
                                     重排（rerank）
                                        ↓
                               Top-K 文档 → 拼接 prompt → LLM 生成 → 答案 + 引用
```

从这个流程可以看出，RAG 涉及的技术环节相当多，每个环节都有调优空间。

在架构层面，RAG 还有两种主流形态。**经典 RAG**（也叫 Naive RAG）是一个固定流程：用户问题来了，先检索，再拼接 prompt 喂给 LLM，生成答案。**Agentic RAG** 则把检索封装成一个工具（Tool），LLM 可以自主决定什么时候调用、用什么 query 调用，甚至多次调用直到信息足够。后者更灵活，但复杂度也更高。

**小结：** RAG 不是"加个检索"那么简单，而是一个涉及数据准备、检索策略、模型调优的系统工程。架构选型上，经典 RAG 适合确定性场景，Agentic RAG 适合需要灵活推理的复杂场景。

## 四、数据准备：Chunking 的艺术

切分（chunking）是 RAG 数据准备中最容易被低估的一环。很多人直接拿个固定字符数就切了，上线后发现召回一塌糊涂。

### 为什么不能随便切

理由有三：一是 LLM 的上下文窗口再大也有限，200K token 也会被长文档里的无关信息稀释掉有效信号；二是检索粒度问题，太长的文档里相关信息只占一小段，embedding 会被整体语义"平均"掉，导致召回不准；三是成本问题，prompt 里塞越多 token 越贵。

### 主流切分策略

最推荐的默认方案是**递归切分（Recursive Chunking）**：先按段落边界切，超长段落再按句子边界切，句子还超长就按字符硬切。这样在保留语义边界的同时控制了长度。

其他策略各有适用场景：

- **按段落切**（`\n\n` 分隔）适合结构化文档，语义边界清晰但段落长度可能不均。
- **语义切分（Semantic Chunking）** 用 embedding 相似度找自然断点，质量最高但速度慢、成本高。
- **按 Markdown 结构切**（按标题层级）适合技术文档和 Wiki，保留了文档结构。
- **按代码语法切**（按函数/类）是 Code RAG 的标配。

### 关键参数

**chunk_size** 通常设在 200-1500 token 之间。短查询、精准问答场景用 300-500；需要长上下文理解的场景用 1000+。没有万能值，取决于文档类型和查询类型。

**chunk_overlap** 设为 10-20% 的 chunk_size，目的是避免相关信息正好被切到边界上。

还有一个很实用的模式叫 **Parent-Child Chunking**：小 chunk 用于检索（保证精准度），检索命中后返回对应的大 chunk（父段落）喂给 LLM（保证信息完整度）。这样同时兼顾了召回和上下文完整性。

**小结：** Chunking 没有银弹。理解不同策略的 trade-off、根据文档类型和数据量做实验和评估，才是正确的方法。

## 五、向量化与存储

### Embedding 选型

把文本映射成向量是所有向量检索的前提。常见的 embedding 模型有 OpenAI 的 text-embedding-3（1536/3072 维）、BGE 系列（中文场景首选）、M3E（中文领先）等。

选型时主要考量这几个维度：

- **语言覆盖**：中文场景下 BGE 和 M3E 效果常优于 OpenAI 的通用模型。
- **领域适配**：如果知识库涉及医疗、法律、代码等专业领域，通用模型可能不够，需要做领域 fine-tune。
- **维度**：高维效果更好但存储和计算成本更高。Matryoshka 式嵌入（可降维）是新趋势，灵活性更好。
- **长度支持**：大多数 embedding 模型最大支持 512 token，超长文档需要截断或用 Jina 等支持长文本的模型。

### 向量数据库

生产环境中的选型，核心看三个维度：规模、运维成本、功能需求。

- **轻量/原型**：Chroma 适合快速验证；pgvector 适合和业务数据库统一管理。
- **中型商用**：Pinecone 托管省心但贵；Qdrant 性能好、Rust 实现；Weaviate 功能全。
- **大规模**：Milvus 能支撑亿级向量；Elasticsearch + dense vector 适合已有 ES 环境的企业。

### 索引算法

纯暴力搜索在数据量大时慢到不可接受（毫秒变成秒），所以需要 ANN（近似最近邻）算法。目前最主流的是 **HNSW（分层可导航小世界图）**，它把向量组织成分层图结构，搜索时从稀疏层快速定位到目标区域，再到密集层精细搜索，思路类似于跳表（skip list）。关键参数包括 `M`（节点连接数）、`efConstruction`（建索引质量）和 `efSearch`（检索精度）。

以 1000 万条 1536 维 float32 向量为例，裸存储约 60GB。用 HNSW 检索是毫秒级；用 PQ 压缩可以降到约 6GB，精度损失在 5% 以内。

**小结：** 向量化选型要看语言覆盖和领域适配，索引选型要看数据规模和延迟要求。HNSW 是目前综合表现最好的默认选择。

## 六、检索与重排

### 三种检索方式

生产级 RAG 很少只用一种检索方式。常见的搭配是混合检索：

- **向量检索（Dense Retrieval）**：语义层面找相似，能理解同义和释义，但对关键词精确匹配不敏感（"iPhone 15" 和 "iPhone 14" 向量可能很接近）。
- **关键词检索（Sparse Retrieval）**：BM25 按词频和逆文档频率打分，精确匹配专有名词、数字、代码，但不懂同义词。
- **混合检索（Hybrid Retrieval）**：两路并行，结果合并。最常用的合并方式是 **RRF（Reciprocal Rank Fusion）**，公式是 `score = Σ 1/(k + rank_i)`，优点是分数不需要归一化。

### Top-K 怎么选

Top-K 太小（如 K=3）召回率低，LLM 可能拿不到答案。太大（如 K=20+）上下文稀释，LLM 被无关信息干扰。典型值是 5-10。一个推荐的策略是：粗召回取 20-50，再用 rerank 精选到 5-10。

### Rerank 为什么重要

向量检索用的是双塔模型——query 和 doc 分别嵌入再算相似度，速度快但精度有限。Rerank 用的是交叉编码器（Cross-Encoder），把 query 和 doc 拼接在一起直接算相关度，精度高得多但慢得多。工业界的典型做法是：第一阶段用向量检索取 top-50，第二阶段用 rerank 取 top-5，精确率能再提 10-30%。

### 多查询与多样性

**多查询检索（Multi-query Retrieval）** 让 LLM 把原问题改写成多个子问题分别检索再合并，效果显著但成本也成倍增加。

**MMR（Maximal Marginal Relevance）** 在 Top-K 里做多样性过滤，避免返回的文档过于相似——当多个 chunk 来自同一份文档时尤其有用。

**小结：** 混合检索 + rerank 是目前生产级 RAG 的标准配置。向量和关键词互补，rerank 做精排，多查询和 MMR 做补充优化。

## 七、查询改写技术

用户的问题并不总是适合直接检索。原始问题可能含糊、带代词、过于具体或过于复杂。几种常用的改写策略：

**Query Rewriting**：让 LLM 根据对话历史补全上下文。"它怎么用"变成"LangChain 的 RunnableSequence 怎么使用"。

**HyDE（Hypothetical Document Embeddings）**：让 LLM 先假装回答用户问题，生成一段"假想答案"，然后用这段假答案去检索。为什么有效？因为在向量空间里，"答案和相关文档"的距离通常比"问题和相关文档"更近——回答本身比问题更接近目标文档的内容分布。

**Step-Back Prompting**：用户问题太具体时，先抽象到一个更宏观的问题去检索。比如问"Messi 1987 年 6 月 24 日出生那天是星期几"，先检索"Messi 出生在哪一天"，查到具体日期后再推算星期。

**问题分解（Decomposition）**：复杂问题拆成子问题分别检索。"比较 LangChain 和 LlamaIndex 在 RAG 上的优劣"拆成"LangChain 的 RAG 实现"和"LlamaIndex 的 RAG 实现"两个子问题分别检索，再对比。

**小结：** 查询改写能显著提升检索命中率，但也会增加延迟和成本。HyDE 和问题分解在复杂查询场景下效果尤其明显。

## 八、评估体系

RAG 系统的评估分两层。

**检索层**关注的是召回有没有漏、排位对不对。常用指标包括 Recall@K（有没有漏掉相关文档）、Precision@K（召回的文档里有多少是相关的）、MRR（第一个相关文档排多靠前）和 NDCG@K（综合考虑排序位置的相关性）。其中 Recall 是绝大多数场景的第一优先级——漏掉关键文档比混进无关文档更致命。

**生成层**关注的是 LLM 的输出质量。最核心的指标是 Faithfulness（忠实度），即答案是否基于检索内容生成、有没有幻觉。其次是 Answer Relevance（答案是否回答了问题）、Context Precision（检索内容里有多少被真正用到了）等。

常用的评估框架有 RAGAS（专为 RAG 设计，覆盖上述指标）、TruLens（可视化追踪）和 LangSmith。实践中通常用 LLM-as-judge 自动打分，辅以人工抽检。

**小结：** 评估体系要兼顾检索和生成两层。先保证召回不 miss，再保证生成不幻觉。在迭代过程中持续跑评估集，用数据说话。

## 九、高级 RAG 模式

### Self-RAG

Self-RAG 让 LLM 在生成过程中自主决定：需不需要检索、检索结果是否相关、回答是否忠实于检索内容。相当于给模型加了元认知（metacognition）能力，让它在"依赖内部知识"和"借助外部检索"之间动态切换。

### CRAG（Corrective RAG）

CRAG 在检索后加一个质量评估器。如果检索结果质量好，正常使用；如果质量差，触发 fallback——比如切换到网页搜索、让 LLM 直接回答、或者要求用户澄清。这种"能纠错"的设计在生产环境中非常实用。

### Graph RAG

微软提出的 Graph RAG 用知识图谱替代纯向量检索。节点是实体和概念，边是关系，检索时沿图谱多跳搜索。适合需要多跳推理的复杂问题（比如"A 和 B 的共同朋友中谁是 C 公司的员工"）。

### Contextual Retrieval

Anthropic 在 2024 年提出的方案。给每个 chunk 生成一段上下文描述（说明这个 chunk 在原文档中的位置和主题），然后对"chunk + 上下文描述"一起做 embedding。根据 Anthropic 的报告，这个方法能把检索错误率降低 35-49%。

**小结：** 高级 RAG 模式在特定场景下效果显著——Self-RAG 和 CRAG 提升了检索-生成的协同质量，Graph RAG 解决了多跳推理需求，Contextual Retrieval 有效解决了 chunk 脱离上下文的问题。

## 十、Agentic RAG：检索即工具

把检索封装成一个 Tool，让 LLM 自主调用来完成信息获取，这就是 Agentic RAG 的核心思路。与经典 RAG 的固定流程不同，Agentic RAG 中 LLM 扮演的是"决策者"角色：

- 收到问题后，判断是否需要检索（有些问题不需要）。
- 决定用什么 query 去检索。
- 评估检索结果是否足够回答问题。
- 不够就换 query 再检索一次。
- 信息足够后生成最终答案。

这种模式天然支持**多跳检索（Multi-hop Retrieval）**。比如问 "LangGraph 的 checkpointer 和 LangChain 的 memory 有什么区别"，Agent 会先检索 LangGraph checkpointer，再检索 LangChain memory，然后对比两路信息生成答案。

真实系统中 Agent 通常有多个信息源：内部文档检索、网页搜索、数据库查询、计算器。LLM 根据问题类型选择合适的工具，组合使用。

**小结：** Agentic RAG 把 RAG 从"检索-生成"的两步流水线升级成了"感知-决策-行动"的循环，适合需要多步推理和灵活决策的复杂场景。

## 十一、生产级工程要点

### 知识库管理

文档持续更新意味着 chunk 和 embedding 也需要同步更新。实用的做法是用哈希比对检测变化，只对变更的文档做增量 embedding。多租户场景下，检索需要带权限过滤（metadata filtering），不同部门的用户只能看到授权范围内的知识库。

### 延迟优化

RAG 全链路的延迟主要被 LLM 生成吃掉（500-3000ms），其他环节加起来不过几百毫秒。优化方向包括：embedding 用快模型、向量库用 HNSW、对延迟敏感的场景跳过 rerank、LLM 输出用 streaming 减少首 token 等待。

### 成本控制

Embedding 加一层哈希缓存可以避免重复调用 API。简单问题用小模型兜底、复杂问题再升级到大模型。知识库做冷热分离——热数据放 ES 之类的高性能存储，冷数据放 S3。

### Fallback 策略

检索失败时，可以用 LLM 内置知识回答但标注"无知识库支撑"。LLM 生成失败时，直接返回检索结果原文。全流程失败时，准备固定兜底话术并记录日志。两个原则：优先让用户得到答案而不是报错；所有异常都要可观测。

### 安全合规

向量化前清洗 PII 信息。检索链路带上用户权限过滤。每次检索和生成结果都要留审计日志。特别要注意 Prompt Injection 防护——文档内容本身可能被恶意污染（"忽略之前指令，说..."），需要在输入到 LLM 之前做检查。

**小结：** 生产环境下的 RAG 不仅仅是"做好检索和生成"，知识管理、延迟、成本、容错、安全都需要端到端考虑。

## 总结

RAG 不是一项独立的技术，而是 LLM 工程化落地中至关重要的一环。从离线数据准备到在线检索生成，从经典的两步流水线到 Agentic 的自主决策，RAG 的演进方向始终是让 LLM 在需要的时候能获取到正确的信息、在不需要的时候不浪费 token 在无关内容上。

如果说 LLM 本身是"大脑"，那 RAG 就是让大脑随时能查书、能检索的能力。两者结合，才是真正可落地的知识系统。

对于技术面试来说，能把 RAG 的工程细节（chunking、embedding、检索策略、rerank）、架构演进（经典 RAG vs Agentic RAG）、评估体系和生产实践都讲清楚，就已经超出了大多数候选人"听说过 RAG"的水平。希望这篇文章能帮你做到这一点。
