# 可观测性基石：日志治理的艺术与工程实践

日志是系统运行时最原始的"黑匣子"。无论是排查线上故障、分析业务数据，还是构建告警体系，日志都是第一手信息源。但日志也是一把双刃剑：打少了查问题像大海捞针，打多了存储成本暴涨且淹没有效信号。

本文从日志怎么打、怎么收、怎么存三个维度，整理一套可落地的日志治理实践。

## 1. 日志级别的正确打开方式

很多团队对日志级别的使用存在随意性。一个常见的错误是：开发人员在 debug 阶段打的日志没有清理就直接上线，或者把正常的业务信息标记为 error，导致半夜告警响个不停。

合理的分级策略应当是：

- **DEBUG：开发环境专用。** 随意打，主要看逻辑执行路径和中间变量。线上环境通常关闭或通过动态开关按需开启。
- **INFO：关键业务流程节点。** 比如订单创建、支付回调到达、状态流转。这条日志记录了"系统做了什么"，是日常排查和业务分析的主力。
- **WARN：异常但可恢复的情况。** 比如外部接口超时后重试成功、配置项缺失后使用默认值。WARN 意味着需要关注，但不需要立即响应。
- **ERROR：需要人工介入的异常。** 比如数据库连接失败、消息消费彻底失败、业务流程因异常中断。ERROR 日志必须带上完整堆栈，让值班人员能快速定位。

**关键原则：** ERROR 级别应当用于"系统无法自行恢复"的场景，而不是用来表达"业务上不期望但逻辑已兜底"的情况。如果代码里 catch 了异常并做了降级处理，那它更可能是 WARN 而非 ERROR。

**小结：** 日志级别的本质是告警过滤的第一道防线。级别分得清，告警才有意义。建议团队制定统一的日志规范并在 code review 中把关。

## 2. 结构化日志：给机器看的艺术

传统的文本日志一行一个格式，用 grep 和 awk 勉强能查。但在微服务架构下，日志量级从每天几百 MB 膨胀到几百 GB，文本日志的查询效率极其低下。

现在的标准做法是**结构化日志**，通常以 JSON 格式输出：

```json
{"level":"INFO","time":"2026-05-03T10:00:00Z","service":"order-service","traceId":"abc123","message":"订单创建成功","orderId":"ORD123456"}
```

这样做有几个好处：
- 字段化存储，可以按任意维度过滤和聚合
- 标准字段（service、traceId）可以自动关联多维度数据
- 与日志收集工具（Filebeat、Vector）天然配合

当然，JSON 日志在开发终端上阅读体验很差。本地开发时可以配置两种输出模式：终端输出用文本格式方便人看，生产环境输出 JSON 格式方便机器处理。

在 Go 生态中，`zap` 和 `zerolog` 是结构化日志的首选。它们的性能大约是标准库 `log` 包的 4-5 倍，核心优化包括避免反射、预分配 buffer、零分配日志对象等。在 Java 生态中，`Logback` 配合 `logstash-logback-encoder` 同样可以输出结构化日志。

**小结：** 结构化日志是将日志从"文本文件"升级为"可查询数据"的关键步骤。建议所有新项目从第一天起就采用结构化日志。

## 3. 日志与链路追踪的关联

日志最大的痛点之一：一个请求跨了多个服务，每个服务各自打日志，出问题时无法串联。

解决这个问题需要引入分布式追踪的 traceId。每个请求在网关层生成一个全局唯一的 traceId，通过 RPC 头部或者 HTTP Header 向下游传递（遵循 W3C TraceContext 标准），所有服务的日志都带上这个 traceId。

这样一来，在日志平台搜索 `traceId=abc123`，就能看到该请求经过的所有服务、每个服务打了什么日志、执行顺序如何。这也是为什么 OpenTelemetry 整合了 Logs、Metrics 和 Traces 三大信号——它们需要共享同一套上下文。

**小结：** traceId 是分布式日志的"胶水"。没有 traceId 的日志在微服务架构下基本等于孤岛。

## 4. 日志采集与存储架构

完整的日志管道通常包含四个环节：

1. **采集（Collector）：** 在业务服务器上部署 Agent 采集日志文件。Filebeat（Elastic 生态）轻量稳定，Vector（Rust 实现）性能更好、支持更丰富的处理场景。
2. **处理（Processor）：** 解析、过滤、清洗日志数据。Logstash 功能丰富但内存占用较高，Fluentd 是 CNCF 项目，与 Kubernetes 集成好。
3. **传输（Transport）：** 将处理后的日志发送到存储后端。需要保证可靠性，避免 Agent 宕机导致日志丢失。
4. **存储与分析（Storage & Query）：** 提供日志的存储和检索能力。

### 存储方案选型

最经典的组合是 **ELK 三件套**：Elasticsearch（存储 + 检索） + Logstash（处理） + Kibana（可视化）。Elasticsearch 的全文搜索能力非常强大，生态成熟，社区资源丰富。

但是 Elasticsearch 并非为日志场景专门设计。它的索引分片机制对日志这种高吞吐、低价值密度的数据来说，存储成本偏高。于是出现了两个方向：

- **ClickHouse：** 列式存储，写入性能极高，压缩比好，存储成本只有 ES 的 1/3 到 1/5。查询能力不如 ES 灵活，但针对日志的时序检索场景足够用。
- **Loki：** Grafana 出品，专为日志设计，不建立全文索引，只对元数据索引。存储成本极低，适合与 Prometheus、Grafana 形成可观测性统一平台。

### 保留策略

结合成本和合规需求，日志保留通常分三层：

- **热数据（7 天）：** 存放在高性能存储上，保证快速查询，用于日常故障排查。
- **温数据（30 天）：** 存放在中等性能存储上，用于短期趋势分析和月度排查。
- **冷数据（归档）：** 超过 30 天的日志压缩归档到对象存储（S3/GCS），仅保留重建现场的能力，不做在线检索。

Elasticsearch 的索引按天滚动是标准做法，避免单个索引过大导致查询和写入性能恶化。

**小结：** 日志存储选型要平衡查询灵活性和存储成本。ELK 适合查询复杂、需要全文检索的场景；ClickHouse 适合高吞吐、低成本写入场景；Loki 适合与 Grafana 生态统一管理的场景。

## 5. 日志安全与规范

日志中不能包含敏感信息。密码、token、身份证号、手机号、银行卡号等个人隐私数据，一律不能出现在日志中。如果确实需要记录，必须做脱敏处理（如 `138****1234`）。

另外几个常见规范：

- **异常日志必须带堆栈：** `logger.error("xxx 失败", exception)`，而不是只打 message。
- **不要在循环中打日志：** 内部调用一秒几万次循环的场景，日志会瞬间打满磁盘。
- **日志打印不要影响业务性能：** 打印日志的代码不应持有锁或在热路径上做耗时的字符串拼接。
- **统一日志格式：** 时间戳统一为 ISO 8601 格式，时区统一为 UTC。

## 总结

日志治理是一个系统工程，从代码中的每一行日志开始，到采集管道，再到存储和查询，每一个环节都需要设计。好的日志体系能让你在故障发生时几分钟内定位问题，坏的日志体系则会让排查变成大海捞针。

从实操角度看，优先做好三件事：**统一日志规范、推进结构化日志、将 traceId 纳入每行日志**。这三件事做好了，后续的采集和存储选型只是技术选择问题。
