# 可观测性三支柱：从 Prometheus 到 OpenTelemetry 的统一演进

早年间做运维，最怕半夜告警电话。打开监控看 CPU 涨了，但不知道是哪个接口导致的；翻日志全是 "INFO: request processed"，根本找不到异常；问下游服务是不是挂了，对方说"我这边一切正常"。

这三个困境恰好对应了可观测性的三根支柱：**指标（Metrics）、日志（Logging）、链路追踪（Tracing）**。它们各自解决了问题的一角，但只有三者联动，才能拼出完整的真相。

## 1. Metrics：从"感觉不对"到"精确量化"

Metrics 回答的问题是："系统现在处于什么状态？"

Prometheus 是这一领域的标杆。它采用 Pull 模型——Prometheus Server 定期从服务暴露的 `/metrics` 端点抓取数据，而不是等服务主动 Push 过来。这个设计看似简单，却解决了 Push 模型下一个致命问题：当监控后端挂了，Push 请求堆积可能直接把业务服务拖垮。Pull 模型下，Prometheus 自己负责采集节奏，业务服务只管暴露数据。

### 四种指标类型

Prometheus 定义了四种指标类型，每种有明确的适用场景：

| 类型 | 特征 | 适用场景 |
|------|------|---------|
| **Counter** | 只增不减 | 请求数、错误数、任务完成数 |
| **Gauge** | 可增可减 | CPU、内存、连接数、队列长度 |
| **Histogram** | 分布统计（按 bucket） | 请求延迟分布、响应大小分布 |
| **Summary** | 分位数统计 | P50/P99 延迟（客户端计算） |

一个常见坑：Histogram 的 bucket 配置。如果你的请求延迟大多在 100ms 左右，但 bucket 只配了 `[10, 50, 100, 500, 1000]`，那么 90% 的请求会落进 `+Inf`，直方图几乎看不到分布细节。应该根据实际延迟分布调整 bucket，比如 `[50, 100, 200, 500, 1000, 2000]`。

### PromQL：监控的 SQL

Prometheus 的查询语言 PromQL 初看有些反直觉，特别是 `rate()` 和 `irate()` 的区别：

```
rate(http_requests_total[5m])   // 5分钟内每秒平均增长率
irate(http_requests_total[5m])  // 最近两个采样点的瞬时增长率
```

`rate()` 适合看趋势（平滑），`irate()` 适合捕获突发（敏感）。做容量规划用 `rate()`，查瞬时毛刺用 `irate()`。

### 告警不是越多越好

告警规则里的 `for` 参数是最容易被低估的配置。`expr: error_rate > 0.05` 加上 `for: 5m` 意味着错误率超过 5% 且持续 5 分钟才触发。没有 `for`，每次指标短暂抖动都是一条告警——很快所有人都会把告警静音，"狼来了"效应让真正的故障被淹没。

良好的告警策略：重要的配 `for` 防抖动，用 Alertmanager 的 inhibition 机制抑制下游告警（数据库挂了就不要为依赖它的每个服务都发一条），silence 留给已知的维护窗口。

**小结：** Metrics 是观测系统的脉搏。Prometheus 的 Pull 模型、四种指标类型、PromQL 和告警体系，构成了从数据采集到响应动作的完整闭环。难点不在配置本身，而在设计出能真正反映系统健康的指标体系。

## 2. Logging：从"大海捞针"到"结构化的数据流"

Metrics 能告诉你"出事了"，但要搞清楚"为什么会出事"，还得看日志。

### 日志级别的纪律

最基础的规范也是最容易出问题的：日志级别。一个实践过的标准：

- **DEBUG**：开发环境随便打，线上默认关
- **INFO**：关键业务流程的节点（订单状态变更、支付回调）
- **WARN**：异常但可自愈（重试成功、降级触发）
- **ERROR**：需要人工介入（支付失败、数据不一致）

最常见的翻车现场是把正常业务信息打成 ERROR——比如"用户输入了错误的验证码"。这不叫 ERROR，这是正常的业务逻辑分支。乱用级别的结果就是半夜被不需要处理的告警叫醒。

### 结构化日志的价值

传统日志是人读的："2024-01-15 10:30:45 User 12345 login success"。但当你有几百万条日志时，人眼看不过来，需要用工具检索和分析。这就是结构化日志的意义：

```json
{"time":"2024-01-15T10:30:45Z","level":"info","msg":"user login","userId":12345,"ip":"10.0.1.5","duration_ms":42}
```

机器可以按 `level=error AND duration_ms>1000` 快速过滤。代价是开发时看 JSON 确实累眼——建议开发环境输出人类可读格式，线上输出 JSON。

性能方面，Go 的 `zap` 和 `zerolog` 比标准库快一个数量级（零分配、无反射），Java 的 `log4j2` 异步追加也能显著降低日志的 IO 开销。

### 日志规范的红线

三条不可妥协的原则：

- **绝不打印敏感信息**：密码、Token、身份证号、银行卡号。日志系统往往比业务系统更容易被内部人员访问。
- **异常必须带堆栈**：`log.error("something failed", e)` 是没用的，要带上完整的调用链。
- **不在循环中打印日志**：哪怕是 `debug` 级别——就算不输出，字符串拼接和参数求值也会执行。

**小结：** 日志的本质是将运行时的"黑盒"变成"白盒"。结构化日志让机器能查询，合理的级别让告警有价值，安全规范则是防止日志本身变成漏洞。日志打得好，排查问题事半功倍；打不好，它就是一堆占磁盘的噪音。

## 3. Tracing：从"单点排查"到"全链路视野"

如果一个请求慢了 500ms，只查一个服务的日志是找不到根因的——因为这个请求可能经过了网关 → 订单服务 → 库存服务 → 支付服务 → 通知服务，延迟到底出在哪个环节？

### OpenTelemetry：三位一体的标准

OpenTelemetry（OTel）是 CNCF 的孵化项目，已经成为分布式追踪的事实标准。它的野心不只是 Tracing——OTel 的设计是 Metrics、Logging、Tracing 三位一体，用一个统一的 SDK 和 Collector 覆盖可观测性的全部需求。

架构上分为四层：

- **API**：定义接口规范，各语言 SDK 实现
- **SDK**：埋点能力的具体实现
- **Collector**：接收、处理（批处理、过滤、采样）、导出数据的中间层
- **Exporter**：将数据发送到后端存储（Jaeger、Prometheus、Grafana Tempo 等）

### 自动埋点 vs 手动埋点

自动埋点几乎零代码侵入：HTTP 服务、gRPC 调用、数据库查询、消息队列——这些标准库的 Instrumentation 开箱即用，自动创建 Span 并传递 Context。

手动埋点用于业务逻辑的精细化观测：

```java
Span span = tracer.spanBuilder("createOrder").startSpan();
try {
  checkInventory();   // span 内会自动记录耗时
  processPayment();
} finally {
  span.end();
}
```

一个好的实践：自动埋点覆盖 80% 的标准链路，剩下 20% 的关键业务节点做手动埋点。

### 采样策略的权衡

全量追踪在流量大的系统里会产生海量数据。采样是必须的：

| 策略 | 原理 | 优点 | 缺点 |
|------|------|------|------|
| 固定采样率 | 10% 的请求被追踪 | 简单 | 异常请求可能被漏掉 |
| 自适应采样 | 正常少采，异常多采 | 异常全保留 | 实现复杂 |
| 头部采样 | 请求开始就决定 | 即时决策 | 可能漏掉变慢的请求 |
| 尾部采样 | 请求结束再决策 | 保留所有慢/错请求 | 需缓存所有 Span |

生产环境推荐自适应 + 尾部采样的组合：正常请求按 10% 采样，但所有超过 1s 或出错的请求 100% 保留。

### 后端选型

Jaeger 曾是 Tracing 的代名词，而新一代的选择是 **Grafana Tempo**——它不建专门的存储集群，而是直接把 Trace 数据存进对象存储（S3/GCS），成本大幅降低，且天然与 Grafana 的可视化能力整合。

**小结：** Tracing 补上了 Metrics 和 Logging 之间缺失的因果链。OpenTelemetry 统一了三大支柱的数据采集标准，让它们不再是三座孤岛。一个好的 Tracing 实践不是装个 Jaeger 就算完事，而是需要设计采样策略、规划埋点粒度、选好存储后端。

## 总结

可观测性不是装几个开源工具就完事，它是一个系统工程：

```
Metrics（Prometheus + Grafana）→ 告诉你"出问题了"
Logging（Zap + ELK/ClickHouse） → 告诉你"为什么出问题"
Tracing（OpenTelemetry + Tempo）→ 告诉你"在哪里出的问题"
```

三者各司其职，又互相补充。一个完整的排查链路应该是：Grafana 面板发现延迟飙升 → 点进去看 Trace，发现订单服务的 `processPayment` Span 耗时 3s → 跳到这个 TraceId 关联的日志 → 看到 `PaymentGateway timeout after 3000ms` → 定位到下游支付网关故障。

最后，几个工程上的核心原则：

- **指标命名要规范**，Label 基数要控制（不要用 userId 做 label）
- **日志必须结构化**，绝不能泄露敏感信息
- **TraceId 要贯穿全链路**，异步调用记得传递 Context
- **告警要有 for 缓冲**，防止抖动变成噪音
- **三大支柱的数据要能关联**：Metrics → Trace → Logs，缺一环就断一链

工具会迭代，但这个三角框架不会变。
